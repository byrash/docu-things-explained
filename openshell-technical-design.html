1. System Overview

OpenShell provides policy-governed sandbox runtimes for autonomous AI agents. Agents operate inside isolated containers where network, filesystem, process, and inference access are enforced by a layered security stack. The platform ships as a single Docker container embedding K3s, a gRPC gateway, and the sandbox supervisor — requiring only Docker on the host.

4. Enforcement Layers (Defense-in-Depth)

OpenShell applies five concentric security layers so that no single bypassed control grants full access. All layers are configured from the same declarative YAML policy.

1 Network Namespace Isolation

Each sandbox gets a dedicated Linux netns with a veth pair. The agent process enters via setns(CLONE_NEWNET). The only route out is through the host side of the veth where the proxy listens. iptables rules REJECT any traffic not addressed to the proxy, providing fast failure and kernel-level audit logging.

2 seccomp (Syscall Filtering)

A BPF filter applied after PR_SET_NO_NEW_PRIVS blocks dangerous syscalls: memfd_create, ptrace, bpf, mount, io_uring_setup, execveat(AT_EMPTY_PATH), unshare(CLONE_NEWUSER), and seccomp(SET_MODE_FILTER). Socket families AF_PACKET, AF_BLUETOOTH, AF_VSOCK, AF_NETLINK are blocked.

3 Landlock (Filesystem Sandboxing)

Landlock LSM (ABI V2) creates a filesystem allowlist. read_only paths get read access; read_write paths get full access. Everything else is invisible. Applied via restrict_self(). If Landlock is unavailable, the sandbox emits an OCSF high-severity finding and can either degrade or refuse to start.

4 HTTP CONNECT Proxy + OPA (L4 Network Policy)

All TCP egress routes through a proxy on the host side of the veth. Each CONNECT is evaluated by an embedded OPA engine (regorus). Rego policy checks who (binary path, SHA256, ancestors), where (host, port), and returns allow/deny/log.

5 L7 Inspection (Application-Layer Policy)

For endpoints with protocol rules (REST, SQL, inference), the proxy terminates TLS on the allowed tunnel and inspects HTTP method, path, and headers. Per-endpoint Rego rules provide fine-grained control. Access presets simplify common patterns.

Policy Scope

Network Rules (L4)

Allow/deny by host, port, binary path, binary hash, ancestors. Supports wildcards and CIDR ranges. Actions: allow, deny, log.

L7 / REST Rules

Per-endpoint HTTP method + path patterns. Protocol types: rest, sql, inference. TLS auto-termination or skip.

Filesystem Rules

Landlock paths: read_only and read_write lists. Everything else is denied.

Process Rules

Entrypoint binary, user/group, environment variables, working directory, privilege settings.

6. Inference Routing

Agents access LLM APIs through a single explicit endpoint: https://inference.local. There is no implicit catch-all interception for arbitrary hosts. The operator configures the backend; the agent never sees real credentials or model names.

How It Works

1

Agent sends request

Agent uses base_url="https://inference.local/v1" with any OpenAI-compatible client. The model and api_key fields are placeholders — they're rewritten.

2

Proxy intercepts CONNECT

The sandbox proxy recognizes CONNECT inference.local:443, responds 200 Connection Established, and enters the inference interception path (not the normal OPA CONNECT flow).

3

TLS termination + pattern detection

Proxy terminates TLS (MITM with sandbox CA), parses the cleartext HTTP request, and classifies it (e.g. openai_chat_completions, anthropic_messages). Unmatched paths get 403.

4

Router resolves backend

openshell-router picks a ResolvedRoute from the cached inference bundle, injects real credentials and model name, and streams the response back through the proxy.

Where OpenShell Stands Out

Strengths

Deepest policy enforcement — only solution with OPA/Rego L4+L7 policy, per-binary network rules, and hot-reloadable policies. Only one with built-in inference routing and privacy-aware credential rewriting. OCSF-grade observability. GPU support via NVIDIA device plugin. Self-hosted = no vendor lock-in or per-sandbox cloud fees.

Weaknesses

Heaviest setup — requires Docker + embedded K3s. Slowest sandbox creation (pod scheduling vs. sub-second VM boot). Alpha maturity. Not natively macOS (runs via Docker). K8s complexity is overkill for single-agent, single-developer workflows.

Key insight: OpenShell is the only solution that combines policy depth (Rego L4+L7, per-binary rules, inference routing) with kernel hardening (Landlock, seccomp, netns). Other solutions optimize for either raw isolation strength (microVMs) or setup simplicity (process wrappers), but none offer the same granularity of runtime policy control. The trade-off is operational complexity and startup latency from the K8s layer.
